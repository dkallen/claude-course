Sandbox Matrix

This map turns the official combinations into something easier to reason about. Read rows as the technical boundary and columns as the approval behavior inside or outside that boundary.

`untrusted`
`on-request`
`never`
`read-only`
Strict browse

Useful when you want inspection plus command gating, but it is less common than plain read-only on-request.

Safe browse

Read files and ask questions. Codex must ask before edits, commands, or network.

CI / quiet

Read-only non-interactive mode. Good for inspection jobs that should never prompt.

`workspace-write`
Guarded edits

Edit locally, but require approval for commands outside the trusted set.

Full-auto preset

The common low-friction local mode: edit in the workspace, ask when the task crosses the boundary.

No prompts

Can be useful in tightly controlled automation, but you are deliberately removing the approval pause.

`danger-full-access`
Broad access

Rarely the right default. If you are already here, ask why the narrower modes are insufficient.

Broad + ask

Still broader than most local tasks require. Approvals remain, but the fence is gone.

Full access

No sandbox and no approvals. This is the true high-trust mode, not the same as `--full-auto`.

Remember

--full-auto is workspace-write plus on-request. It is not unrestricted access.

Boundary Details

Network access, writes outside the workspace, and protected areas like .git/ can still trigger approval or require explicit opt-in.

Default Habit

Start narrow. Expand only when the task proves it needs more room.
