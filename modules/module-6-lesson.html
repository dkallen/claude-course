Module 6: Test-Driven Development with Claude Code

From Concept to Delivery with Claude Code — A Practitioner's Course

Why This Module Matters

With AI, tests stop being just a safety net. They become the cleanest way to specify what you mean. A prompt can still leave room for interpretation. A failing test is much harder to misunderstand.

That is why TDD matters more with Claude than it might in ordinary solo coding. Tests are not just post-hoc verification. They are the contract that tells Claude what success looks like in executable form.

1. Start From Behavior, Not Implementation

The mistake is asking Claude to write tests for the code it is about to write. That often produces implementation-shaped tests: tests that mirror the structure Claude invented instead of validating the behavior you actually want.

The correct sequence is:

  1. state the requirement in plain English
  2. turn that requirement into tests
  3. review the tests critically
  4. only then ask Claude to implement the code that makes them pass
If the tests are weak, the implementation can be perfectly wrong and still pass.

2. Red, Green, Refactor

The loop is simple:

  • Red: write a test that fails for the right reason
  • Green: implement the smallest change that makes it pass
  • Refactor: improve the design while keeping the tests green

Claude can help in all three stages, but each stage needs a different instruction style. During red, you want precision about behavior. During green, you want the smallest coherent implementation. During refactor, you want cleanup without behavioral drift.

3. Why Failing First Matters

If you do not watch the test fail before implementation, you lose the strongest signal in the loop. A passing test might be trivial, disconnected, or pointed at the wrong thing.

A good red phase proves two things at once:

  • the test is actually connected to the behavior
  • the implementation does not exist yet

4. Claude Can Write Weak Tests

AI-assisted TDD goes bad in predictable ways. The biggest danger is not broken syntax. It is false confidence.

Common weak-test patterns

  • assertions so vague that almost any output passes
  • tests that mirror the implementation structure instead of the behavior
  • happy-path-only coverage with no edge cases
  • over-mocking that proves interactions but not outcomes
  • tests that already pass before the feature exists

This is why the review step belongs between “generate tests” and “generate implementation.”

5. Tests as Specification

When Claude writes code from tests, the tests narrow the space of interpretation. Instead of asking Claude to infer your intent from prose alone, you give it examples of acceptable and unacceptable behavior.

That is the real leverage. TDD makes the AI less free to hallucinate the target.

6. Unit Tests vs. Integration Tests

Do not ask for every kind of test at once. Choose the level that matches the risk.

  • Unit tests: best when the risk is in isolated logic and fast feedback matters
  • Integration tests: best when the real risk is the interaction between components, layers, or services

If the bug only appears when several pieces work together, unit tests alone can give you a false sense of safety. If the behavior is pure logic, integration tests may be too slow and vague.

7. What to Ask Claude For

A strong TDD prompt usually asks Claude to:

  1. turn requirements into tests only
  2. avoid implementation until the tests are reviewed
  3. make the initial tests fail for clear reasons
  4. implement the smallest change to go green
  5. refactor only after the behavior is protected

The critical move is splitting the work into stages instead of saying “build the feature and add tests.”

8. What Good Looks Like

A strong TDD artifact set for an AI-assisted change usually includes:

  • a short behavior spec in plain English
  • tests that fail before implementation exists
  • a minimal implementation that goes green
  • a refactor pass that improves structure without changing behavior

If any of those are missing, the cycle is weaker than it looks.

Check

If you can turn requirements into failing tests first, reject weak tests before implementation, and drive Claude through red, green, and refactor deliberately, you are ready to make TDD a real part of your workflow.
