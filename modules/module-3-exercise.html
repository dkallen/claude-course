Module 3: Guided Exercise

The Art of the Prompt — From Ramble to Precision

Goal: Compare what Claude Code does with a loose prompt versus a structured one for the same real task.

Time estimate: 20–30 minutes

Part 1: Pick a Real Task

Choose a task that is real but low risk. Good examples:

  • refactor a duplicated helper
  • write or improve a README
  • add a focused test for an existing behavior
  • clean up a small validation or error-handling flow

Part 2: Use a Loose Prompt

Ask Claude the way you would ask a colleague casually.

This form logic feels messy. Can you clean it up?

Watch what happens. Did Claude choose a sensible scope? Did it make assumptions you didn't explicitly authorize? Did it solve the right problem?

Part 3: Tighten the Prompt

Now re-run the task with a stronger structure:

  • Goal: what should change
  • Context: which files, errors, or patterns matter
  • Constraints: what must not change
  • Done when: how you will judge the result
Goal: centralize validation for the signup flow. Context: current rules are split across the form component and a helper. Constraints: preserve UI, no new dependencies, keep the change narrow. Done when: validation rules live in one place and affected tests pass.

Part 4: Compare the Outputs

  1. Which response was closer to what you actually wanted?
  2. Did the structured prompt reduce scope drift?
  3. Did Claude verify its work more effectively when done-when was explicit?
  4. Was the loose prompt still useful for exploring the problem first?

Part 5: Reflection

Write down one sentence for each of these:

  • When loose prompting helped
  • When loose prompting became risky
  • What constraint most improved the result
  • What you now include by default in higher-stakes prompts

Module 3 Check

Can you deliberately change your prompt style based on whether the task is exploratory or execution-heavy? If yes, move on.
