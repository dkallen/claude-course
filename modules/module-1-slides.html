What Claude Code Actually Is

Before you can use Claude Code well, you need an accurate mental model of what it is. Not the marketing pitch — the actual mechanics.

Most frustration with Claude Code comes from expecting it to be something it isn't. This module fixes that.

Four concepts. Thirty minutes. A foundation for everything that follows.

01

The Agentic Loop

Claude Code isn't a chatbot that happens to write code. It's an agent — a system that can perceive its environment, form plans, take real actions, and learn from the results.

This loop is the engine behind everything Claude Code does.

Read → Plan → Act → Verify

Read
Gather context.
Files, code, your prompt.
→
Plan
Form an approach.
Weigh options.
→
Act
Execute changes.
Real files, real commands.
→
Verify
Check the result.
Errors? Loop back.

The loop is self-correcting — Claude reads errors and retries. But it refines within its current approach. If the approach itself is wrong, it needs you to redirect.

The Loop in Practice

You ask: "Add input validation to the signup form."

ReadScans your project. Finds the signup form. Notices you already use Zod.
PlanDecides to use Zod for validation. Identifies fields and rules needed.
ActWrites the schema, integrates it, updates error handling.
VerifyChecks for syntax errors. Runs tests. Finds a missing import — loops back.

The missing import triggers another micro-cycle: read the error, plan the fix, add the import, verify again. This happens automatically.

02

Environments

Claude Code runs in several places. The model is the same everywhere — what changes is the context surface: what Claude can see and act on.

Four Environments, One Model

Terminal CLI

Purest form. Full filesystem access, shell commands, MCP servers. Maximum control, text only.

VS Code Extension

Same capabilities, richer context. Sees your open files, cursor position, recent edits.

Desktop App (Cowork)

Sandboxed Linux environment. Accessible to non-terminal users. Supports MCP connections.

Web (claude.ai)

Cloud sandbox. No local access. Good for quick tasks, but missing your project context.

The key insight: Choose the environment that gives Claude the context it needs for the task at hand.

The Context Surface Spectrum

More context surface = Claude has more to work with. Less context surface = Claude has to guess more.

Web (generic sandbox) Desktop (your folder) CLI / VS Code (your full env)

This isn't about intelligence — it's about access. The same model, given more context about your specific project, produces dramatically better results.

03

Context Windows

Claude Code has a large but finite working memory. When it fills up, older details get compressed. This is why Claude sometimes "forgets" — and understanding it changes how you structure your work.

The Whiteboard Analogy

Think of the context window as a whiteboard. Claude can write and read everything on it. But the whiteboard has a fixed size.

When it runs out of space, it erases the oldest content — keeping a brief summary, but losing the details. This process is called compaction.

Compaction is automatic and lossy. Specific instructions from early in a conversation can get summarized into vague generalities — or lost entirely.

This is mechanical, not careless. Claude literally can't see the earlier content anymore.

Working With Compaction

→Prefer short, focused sessions over marathon conversations. Each new session starts with a fresh context window.
→Front-load the important stuff. Instructions at the start of a long conversation are most vulnerable to being compacted away.
→Use CLAUDE.md for critical context (Module 2). It gets re-read every session, so it survives compaction.
→Watch for the signs: contradicting earlier decisions, repeating work, re-asking answered questions. That's your cue to start fresh.
04

The Jagged Capability Frontier

This is the most important concept in this module. Claude Code's capabilities aren't uniformly good or bad — they're jagged. Spectacularly good at some things, surprisingly poor at others, with an unintuitive boundary between the two.

Where Claude Code Excels

●Understanding existing code — "What does this do?" gets reliably clear, accurate answers.
●Boilerplate and standard patterns — CRUD, APIs, components, config. Efficient and accurate.
●Refactoring with clear intent — Rename, extract, convert. Mechanical transformations.
●Multi-file coordination — Understands how files relate. Makes coordinated changes without losing track.
●Documentation generation — Synthesizes code into readable docs, READMEs, changelogs.
●Debugging from error messages — Stack trace + code access = remarkably effective diagnosis.

Where Claude Code Struggles

○Novel algorithms — Can't be pattern-matched. Looks right, often has subtle bugs.
○Long-session consistency — Compaction causes contradictions with earlier decisions.
○Knowing what it doesn't know — Confidence is always high, even when it's guessing.
○UI/UX judgment — Functional but rarely beautiful. Optimizes for "works" not "feels right."
○Niche domains — Falls back on generic patterns with unusual frameworks.
○Security edge cases — May miss subtle vulnerabilities. Always get specialist review.

When to Trust Claude Code

Four questions to calibrate your trust for any given task:

1.Is this a common pattern? Yes → high trust.
2.Does this require judgment or taste? Yes → verify carefully.
3.Can I easily verify the output? Yes → let Claude try, then check.
4.What's the cost of an error? Low → let Claude run. High → verify everything.

Work with the grain: give Claude tasks where it's strong, handle the weak spots yourself, and always verify near the jagged edge.
