Module 3: The Art of the Prompt — From Ramble to Precision

From Concept to Delivery with Claude Code — A Practitioner's Course

Why This Module Matters

Claude Code is unusually forgiving. You can describe a problem the way you'd explain it to a smart colleague over coffee and often get something useful back. That is a real strength. It is also a trap.

If you stay in coffee-chat mode for tasks that need precision, Claude will fill in gaps you never meant to leave open. Sometimes it will fill them brilliantly. Sometimes it will confidently optimize for the wrong thing. This module teaches you to control that dial on purpose.

1. Claude Is Good at Interpreting You

Claude Code is not a brittle parser. It is good at reading intent from messy human language. That is why prompts like this can work surprisingly well:

I need to clean this feature up. It mostly works, but the flow feels messy and I think the validation logic is spread around in too many places.

For exploratory work, this is often enough. Claude can inspect the code, infer the likely problem, and propose a sensible direction.

The danger is that this success teaches you the wrong lesson. It can make you think prompt precision doesn't matter. It does. It just matters differently depending on the task.

2. Loose vs. Tight Prompting

Think of prompts on a spectrum.

Loose prompts

Loose prompts are good for exploration, diagnosis, idea generation, and cases where you genuinely want Claude to help define the shape of the problem.

  • good when the task is fuzzy
  • good when you want options
  • good when you want Claude to challenge your framing

Tight prompts

Tight prompts are better when you already know what success looks like and you want Claude to execute within clear bounds.

  • good when the work is high-stakes
  • good when scope drift is dangerous
  • good when the result must be easy to review and verify

The mistake is not using one or the other. The mistake is using the wrong one for the job.

3. Director Mode

One of the most useful prompt styles is what we can call Director Mode: describe the outcome, the constraints, and the standard, but do not micromanage every keystroke.

Instead of:

Open this file, then move this function, then rename this, then add these tests, then update this import...

You say something more like:

Refactor the signup validation so the rules live in one place, the UI stays unchanged, and the result is covered by focused tests. Keep the change narrow and use existing patterns in the repo.

This is usually better because Claude is good at handling the local mechanics once the destination is clear. You stay in charge of the outcome. Claude handles the route.

4. The Four-Part Prompt Structure

When you need more control, a strong default structure is:

  1. Goal: What are we trying to change or build?
  2. Context: What files, errors, examples, or constraints matter?
  3. Constraints: What must Claude preserve, avoid, or follow?
  4. Done when: What must be true before you consider the task finished?

That structure does two things at once:

  • it reduces guesswork
  • it makes the output easier to review

A surprising amount of prompt quality comes down to adding the final part. Many weak prompts say what to do but never define what finished work should look like.

5. Constraints, Examples, and Anti-Examples

Constraints

Constraints keep Claude from solving the wrong problem. Examples:

  • do not add new dependencies
  • preserve the current UI
  • keep the change under a few files if possible
  • use the existing test runner

Examples

Examples are useful when there is a pattern you want Claude to imitate. A good example teaches tone, shape, and expectations faster than abstract instructions do.

Anti-examples

Anti-examples are underrated. Sometimes the fastest way to clarify the prompt is to say what not to do:

  • do not redesign the architecture
  • do not rewrite unrelated files
  • do not give me generic best practices; inspect this repo first

6. Effort Levels and "Ultrathink"

Some tasks need more reasoning than others. A straightforward rename or boilerplate change does not need the same cognitive budget as a bug hunt across multiple files or a tradeoff-heavy refactor.

This is where effort control matters. If the task is simple, extra thinking can just slow things down. If the task is complex, under-thinking leads to shallow pattern matching and premature action.

The practical lesson is simple:

  • lower effort for routine, well-scoped work
  • higher effort for debugging, architecture, ambiguity, or multi-step planning

The point is not to always ask Claude to think harder. The point is to match the thinking budget to the task.

7. Common Prompting Failure Modes

Failure mode 1: vague task, specific expectations

You ask loosely, but judge tightly. That mismatch creates frustration. If you care about specific constraints, put them in the prompt.

Failure mode 2: over-specified mechanics

You hand Claude an exact implementation path when the real need is just a clean outcome. This can make the result brittle and prevent Claude from using better local knowledge from the codebase.

Failure mode 3: missing success criteria

Claude does something plausible, but you never told it how to know it was done. This invites partial solutions and weak verification.

Failure mode 4: using exploration prompts for execution work

"Take a look and improve this" is fine for discovery. It is not fine for a high-confidence production fix.

8. A Practical Heuristic

Before you send a prompt, ask four quick questions:

  1. Am I exploring or executing?
  2. What context does Claude actually need?
  3. What must not change?
  4. How will I know this is done?

If you can answer those, your prompt quality improves immediately.

Tying It Together

Prompting Claude Code well is not about ritual phrasing. It is about matching your prompt style to the kind of work you are doing. Loose prompts are good for discovery. Tight prompts are good for delivery. Director Mode is often the sweet spot: clear outcome, clear constraints, enough room for Claude to use judgment.

As the task gets riskier, move from implication to explicitness. Add constraints. Add done-when. Add examples if the pattern matters. Raise effort only when the problem deserves it.

Check

Can you calibrate your prompting style to the complexity of the task — loose for exploration, tight for execution?
