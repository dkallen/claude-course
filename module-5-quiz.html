Module 5: Scenario Quiz

Use these questions to test whether you can distinguish architecture work from implementation detail.

1. Which statement is architecture?

  • A. Use hooks in every form file
  • B. Keep validation rules owned by a single domain component and consumed by the UI

Answer: B. It defines a responsibility boundary, not a local coding tactic.

2. What is the clearest sign a component boundary is weak?

  • A. The component has a short name
  • B. Business rules are duplicated on both sides of the boundary

Answer: B. Duplication usually means the ownership split is unclear.

3. Why keep first-pass architecture coarse?

  • A. To avoid pretending detail decisions are already settled
  • B. Because detail is never useful

Answer: A. Early detail often hides unresolved structure questions.

4. Which is the better interface question?

  • A. What CSS classes should this component use?
  • B. What promises does this component make to the rest of the system?

Answer: B. Interfaces are commitments across boundaries.

5. If the design discussion includes no tradeoffs, what is the likely problem?

  • A. The system is unusually simple
  • B. The discussion is still too vague to be real architecture

Answer: B. Real architecture always makes some things easier and some harder.

6. What should the output of this stage be?

  • A. A short architecture brief that constrains implementation
  • B. Every class, endpoint, and table finalized

Answer: A. You want enough structure to guide the build, not false precision.
