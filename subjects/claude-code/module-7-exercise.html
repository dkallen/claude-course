Module 7: Guided Exercise

Write a skill for a task you repeat. Then make it trigger automatically.

Goal: Build a working personal skill, test manual invocation, and confirm auto-invocation works.

Time estimate: 30–45 minutes

Part 1: Choose the Right Task

Think of a task you have explained to Claude more than twice — a commit message convention, a PR review checklist, a deployment sequence, a documentation format. It should be specific enough to write clear instructions for and broad enough to use again.

Avoid tasks that are already one-liners. Skills have overhead. The payoff comes from procedures that are annoying to retype, not trivial ones.

Part 2: Create the Skill Directory

Create a personal skill so it is available across all your projects. Replace my-skill-name with a lowercase, hyphenated name.

mkdir -p ~/.claude/skills/my-skill-name

Part 3: Write SKILL.md

Create ~/.claude/skills/my-skill-name/SKILL.md. Use the template below as a starting point.

---
name: my-skill-name
description: [One sentence: what it does and when Claude should use it.
  Include the phrases a user would naturally say that should trigger it.
  Example: Use when writing commit messages, summarising changes,
  or when asked to prepare a git commit.]
disable-model-invocation: false
---

[Your instructions here. Write in imperative form: "Always include...",
"Start with...", "When X, do Y".]

[If the task has steps, number them.]

[If there are things Claude should never do, say so explicitly.]

A worked example for a commit-message skill:

---
name: commit-message
description: Writes a conventional commit message from staged changes.
  Use when committing code, summarising a diff, or asked to write a
  git commit message.
disable-model-invocation: false
allowed-tools: Bash(git diff *) Bash(git status *)
---

Write a conventional commit message for the staged changes:

1. Run `git diff --cached` to read what is staged
2. Choose the correct type: feat, fix, docs, refactor, test, chore
3. Write a subject line under 72 characters: `type: short description`
4. Add a blank line, then 1–3 sentences explaining WHY, not WHAT
5. Do not mention file names in the subject line
6. Do not use vague verbs like "update" or "change" — be specific

Part 4: Test Manual Invocation

Open Claude Code in any project. Run the slash command directly:

/my-skill-name

Observe what Claude produces. Then run it two more times with different inputs or contexts. Look for:

  • Did it follow your instructions literally?
  • Did it miss a step or interpret something loosely?
  • Is there anything in the output you would never want?

Revise SKILL.md for anything that went wrong. You do not need to restart Claude Code — skill file changes are picked up in the current session.

Part 5: Test Auto-Invocation

Now describe your task without using the slash command. Use natural language that matches the trigger phrases in your description.

If your skill is the commit-message example, try:

  • "Can you write a commit for these changes?"
  • "Prepare a git commit message for what I just did"
  • "Summarise these staged changes as a commit"

Claude should invoke the skill automatically. If it does not, your description needs stronger keywords. Strengthen it and try again.

If you want to prevent auto-invocation entirely, add disable-model-invocation: true to the frontmatter.

Part 6: Add a Supporting File (Optional)

If your skill references detailed material — an API spec, a style guide, a list of conventions — move that material to a separate file in the skill directory rather than bloating SKILL.md.

~/.claude/skills/my-skill-name/
├── SKILL.md
└── conventions.md     # loaded only when needed

Reference it from SKILL.md:

For the full conventions list, see [conventions.md](conventions.md).

Part 7: Capture What You Learned

Before moving on, note:

  • What did you have to rewrite in SKILL.md after the first test run?
  • Did auto-invocation trigger when you expected it to?
  • Is there a second task in your workflow that deserves its own skill?

Module 7 check: Does your skill produce consistent output across different inputs without re-explanation? Does it trigger automatically when relevant and stay quiet when it should not? If yes, you are ready for Module 8.
