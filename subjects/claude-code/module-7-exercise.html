Module 7: Guided Exercise

Write a skill from scratch, place it correctly, and test it until you trust it.

Goal: Build one working skill and verify that it behaves reliably across different inputs.

Time estimate: 30–45 minutes

Part 1: Browse Before You Build

Open Claude Code and type /. Spend two minutes reading through the built-in skills in the palette. Pick one and look at what it does when invoked.

Notice: What does the description tell you before you run it? How specific is the instruction? Does it use $ARGUMENTS?

You're not looking for anything in particular — just orienting yourself to what a well-formed skill feels like.

Part 2: Choose a Task Worth Encoding

Think of something you've asked Claude to do more than twice in the same general way. It doesn't need to be complex. Good candidates:

  • Writing a commit message in a consistent style
  • Explaining what a file or function does
  • Checking code for a specific category of issue (security, performance, style)
  • Generating a structured summary of recent changes
  • Drafting a PR description from staged changes

If nothing comes to mind, use this one: Write a commit message for the current staged changes. Format it with a subject line under 72 characters, followed by a blank line and 2–3 bullet points explaining what changed and why.

Part 3: Draft the Skill File

Use the Skill Builder in this module, or draft the file directly. Either way, create a Markdown file with:

  • A clear H1 description (what someone sees in the palette)
  • A specific instruction body (what Claude actually does)
  • A $ARGUMENTS placeholder if the skill takes variable input

Example structure:

# Write a commit message for the current staged changes

Examine the staged diff and write a commit message in this format:

Subject: A clear summary under 72 characters (imperative mood)

- What changed and why (one bullet per distinct change)
- Omit obvious details visible in the diff itself
- Focus on intent, not mechanics

Save the file in .claude/commands/ (or ~/.claude/commands/ if it's a personal habit). The filename is the command name — no spaces, use hyphens.

Part 4: Invoke It Three Times

Open a Claude Code session in a project and invoke your new skill three times with different context:

  1. Typical case: The kind of input you imagined when writing the skill
  2. Unusual case: A larger change, a smaller change, or a different kind of file
  3. Stressful case: Ambiguous input, a file you don't normally work with, or very minimal content

For each invocation, ask yourself: did the output match what I actually needed, or did the instruction leave room for Claude to interpret things in a way I didn't want?

Part 5: Refine the Instruction

Based on the three tests, identify the weakest part of your instruction body. Common problems:

  • Too vague: Claude interpreted "review" differently than you meant
  • Too broad: Claude tried to cover everything when you only needed one thing
  • Missing constraints: Output format was inconsistent between invocations
  • Wrong placement: The skill is general-purpose but you put it in a project-specific location

Edit the file. Invoke it again. Is it better?

Part 6: Capture What You Learned

Write a short note (doesn't need to be formal) answering these questions:

  • What was vague in your first draft that you had to fix?
  • Did the skill behave differently across the three test cases, and if so, how?
  • Is this a project-level skill, a user-level skill, or both? Where did you put it and why?
  • What would make this skill better six months from now when your project has evolved?

This reflection is the real artifact. A skill file you write in 10 minutes is the start — this note is how you turn it into something durable.
