Naming Convention

  • Lowercase, hyphens only — no spaces
  • Name should describe the action, not the tool: /review-pr not /claude-review
  • Short enough to type from memory
  • Distinct enough that two skills don't feel ambiguous side by side in the palette

Testing a Skill

  1. Invoke on a typical case — does the output match intent?
  2. Invoke on an unusual case — different file type, larger scope, missing input
  3. Invoke without $ARGUMENTS if it's optional — does it degrade sensibly?
  4. Check that the description in the palette is clear enough to remember without reading the file

Design Principles

DoAvoid
One skill, one jobSkills that try to do 3 things — they fail one inconsistently
Concrete instruction bodyVague verbs like "review" or "improve" without specifying what good looks like
Project-level for codebase knowledgeMixing personal habits with project conventions in the same file
Browse built-ins before buildingDuplicating a built-in skill that already does the job well
