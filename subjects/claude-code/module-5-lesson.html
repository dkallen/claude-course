Module 5: Architecture — Designing Before Building

From Concept to Delivery with Claude Code — A Practitioner's Course

Why This Module Matters

Once the idea is clear, the next failure mode is building too soon. Teams jump from a good brief straight into implementation and then discover, in code, that nobody agreed on structure, boundaries, or tradeoffs. Architecture is the stage where that confusion becomes visible before it becomes expensive.

Claude Code is useful here because it can help you decompose a system, compare options, and expose assumptions. It is not useful if you ask it to produce architecture theater: boxes with no responsibilities, layers with no purpose, or diagrams that do not constrain implementation.

1. Architecture Is About Responsibility

The first architecture question is not “what technology should we use?” It is “what responsibilities exist, and how should they be separated?”

If responsibilities are unclear, everything else becomes premature. The right database does not save a design where business rules, orchestration, state management, and UI concerns are mixed together.

Good architecture starts by deciding what each part of the system is allowed to own.

2. Start From the Brief, Not From the Stack

Your Module 4 brief should drive the architecture. Goals tell you what matters. Problems tell you where pressure exists. Risks tell you where the design must be defensive. Scope tells you what not to optimize for yet.

If you start with the stack instead, architecture turns into preference projection. You end up selecting tools before you have named the shape of the work.

3. Components Need Crisp Jobs

A component is only useful if you can describe its job in one sentence. “Backend,” “frontend,” or “service layer” is not enough. The useful question is: what responsibility lives here that should not live elsewhere?

  • where is state created, stored, and updated?
  • where do business rules live?
  • where does orchestration happen?
  • where are external dependencies isolated?

If the same rule appears in three places, architecture has already failed.

4. Interfaces Are Commitments

Every boundary creates an interface, even if it is informal. The interface might be a function contract, a data shape, an event, or a network endpoint. The point is the same: one part of the system is making promises to another.

Bad architecture leaves those promises vague. Good architecture names the important ones early enough that implementation can align around them.

5. Architecture Is Mostly Tradeoffs

There is no architecture without tradeoffs. Simplicity may reduce flexibility. Isolation may add coordination cost. Speed of delivery may narrow future options. If the design discussion contains no tradeoffs, it is probably still operating at slogan level.

Claude is valuable here because it can force explicit comparison:

  • what does this design make easier?
  • what does it make harder?
  • what risks does it reduce?
  • what complexity does it introduce?

6. Keep the First Pass Coarse

Early architecture should be coarse-grained. You are defining major components, major flows, and major constraints. If you jump immediately into class diagrams, exact schemas, or file-level organization, you are often using detail to avoid uncertainty.

The first pass should be simple enough to explain aloud and strong enough to guide implementation choices.

7. What Claude Helps With

  • proposing decompositions from a written brief
  • challenging component boundaries
  • comparing architecture options against the stated goals
  • surfacing hidden interfaces and coordination costs
  • turning rough diagrams into a reusable architecture brief

The main failure mode is letting Claude generate structure without pressure. Make it justify every major boundary.

8. A Good Output for This Stage

Before coding begins, you want a short architecture brief that names:

  1. the core components
  2. the responsibility of each component
  3. the key interfaces between them
  4. the main tradeoffs and open questions
  5. the risks the design is trying to manage

That is enough to constrain implementation without pretending every detail is settled.

Check

If you can turn a refined idea into a structure with named responsibilities, clear boundaries, and explicit tradeoffs, you are ready to move into implementation discipline.
