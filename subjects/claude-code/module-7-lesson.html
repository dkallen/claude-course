Module 7: Skills — Reusable Intelligence

From Concept to Delivery with Claude Code — A Practitioner's Course

Why This Module Matters

Every time you tell Claude how to write a commit message, how to format a PR description, or how to structure a code review, that knowledge evaporates when the session ends. The next session, you explain it again. And the session after that.

Skills break this cycle. A skill captures an instruction once — in a file — and makes it invocable with a slash command. The expertise lives in the repository. Claude doesn't need to be briefed again.

1. What a Skill Is

A skill is a Markdown file in .claude/commands/. That's it. The filename becomes a slash command. The file content is the instruction Claude executes when you invoke it.

.claude/
  commands/
    commit.md          →  /commit
    review-pr.md       →  /review-pr
    explain-error.md   →  /explain-error

There are two places skills can live:

  • Project-level: .claude/commands/ in your project root — version-controlled, shared with anyone who works in that repo
  • User-level: ~/.claude/commands/ — personal, applies to every project you open
Project-level skills encode how this codebase works. User-level skills encode how you work.

2. Anatomy of a Skill File

A skill file has three parts: a description, an instruction body, and an optional arguments placeholder.

# Summarize the changes in the current branch as a PR description

Write a clear pull request description for the staged or committed
changes in this branch. Structure it with:

- A one-line summary (under 70 characters)
- A "What changed" section with 2–4 bullet points
- A "Why" section explaining the motivation

Base the description on the actual diff, not assumptions.
$ARGUMENTS

Breaking that down:

  • The H1 line is the description shown in the command palette when you type /
  • The body is the full instruction Claude reads and follows
  • $ARGUMENTS is replaced at runtime with whatever you type after the command name

3. The $ARGUMENTS Placeholder

Without $ARGUMENTS, a skill runs identically every time — a fixed instruction. With it, you can pass context at invocation time.

# Explain what this function does and why it might fail

Look at the function named $ARGUMENTS and explain:
1. What it does, in plain English
2. What inputs might cause unexpected behavior
3. Any edge cases worth testing

Invocation: /explain-function parseUserInput

$ARGUMENTS becomes parseUserInput. The skill now knows which function to target.

You can also use $ARGUMENTS mid-sentence, or use it to accept free-form instructions that modify the skill's default behavior.

4. The Command Palette

Type / in any Claude Code session and the command palette appears. It shows all available skills: built-in Claude Code skills, your user-level skills, and the project-level skills for the current directory.

The palette is organized by source. It also shows the first-line description for each skill, which is why that H1 line matters — it's your skill's label in the UI.

If your skill description is vague, the palette is vague. Write the description like you're labeling a button someone else will press.

5. Built-in Skills

Claude Code ships with built-in skills for common tasks. Browse them by typing / in a fresh session. Before writing your own, check whether a built-in already does what you want — or comes close enough that you only need to extend it.

Studying the built-in skills is also useful for understanding what good skill design looks like:

  • Short, unambiguous command names
  • First lines that function as clear labels
  • Focused instruction bodies — one skill, one job
  • Deliberate use of $ARGUMENTS only when input actually varies

6. Project vs. User Skills — When to Use Each

Use project-level when...Use user-level when...
The skill is specific to how this codebase is structuredYou want the skill across all projects
You want teammates to have the same skillThe skill encodes your personal style or workflow
The skill should be version-controlled with the repoThe skill is an IDE-like habit, not project knowledge

Some skills live in both. A general "write a PR description" might be user-level. A "write a PR description following this project's contribution guidelines" should be project-level.

7. Skill Composition

Skills compose naturally because they are just instructions. A higher-order skill can invoke the behaviors from other skills by describing the same behavior in its instruction body. There is no special syntax for calling one skill from another — you describe the workflow, and Claude executes it.

When to compose:

  • You have two small, focused skills that often run back-to-back
  • You want a "meta" skill that orchestrates several steps in sequence
  • A skill is growing complex enough that splitting it into parts makes each part testable independently

When not to compose:

  • The combined skill would be so long it obscures what it does
  • The steps are already fast enough to run manually in sequence
  • The shared steps are different enough in each skill that one description can't cover both

8. Writing Skills That Work

The most common failure mode is writing a skill that works in the specific case you imagined but breaks when the inputs change slightly. Three practices help:

Be concrete, not abstract

Vague: "Review this code." Concrete: "Review this code for off-by-one errors in loop bounds, null dereferences on user-supplied input, and SQL injection in any string interpolated into a query. For each issue found, show the line, explain the risk, and suggest a fix."

Test with atypical inputs

Run the skill on something other than the case you wrote it for. Does it still behave sensibly when the codebase is different, or when the argument is a partial name, or when the diff is large?

One skill, one job

Skills that try to do too much fail inconsistently. A skill that writes commit messages and runs tests and updates the changelog will sometimes do all three and sometimes do one. Split the jobs. Keep each skill small enough that its success condition is clear.

9. Testing Skills Deliberately

The test for a skill is not "did it run." It is "did it produce the right output across different inputs."

  1. Invoke it on a typical case. Is the output what you expected?
  2. Invoke it on a boundary case. Does it still make sense?
  3. Invoke it without $ARGUMENTS if the argument is optional. Does it degrade gracefully?
  4. Wait two weeks. Come back and invoke it. Does it still do what you need, or has the project evolved past it?

Iteration is normal. The first version of a skill is a draft. The real version emerges after a few real uses.

Check

If you can write a skill file, place it in the right location, understand how $ARGUMENTS works, and test it with inputs that would expose a weak instruction body — you're ready to move on. The builder in this module will help you draft one. The exercise will help you test one.
