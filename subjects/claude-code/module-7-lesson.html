Module 7: Skills — Reusable Intelligence

From Concept to Delivery with Claude Code — A Practitioner's Course

Accuracy note: Claude Code evolves quickly. This lesson sticks to behavior Anthropic documents publicly today, and it calls out the places where settings, permissions, or invocation mode change what Claude actually sees.

Why This Module Matters

Every time you paste the same playbook into a chat, you are doing work Claude could remember. Every time a section of your CLAUDE.md has grown into a step-by-step procedure, you have created something that deserves its own home.

Skills are that home. A skill is a directory containing a SKILL.md file that packages a playbook for Claude to apply on demand — automatically when relevant, or on command when you type /skill-name. Write it once. Stop re-explaining it forever.

1. The Three Instruction Layers and When to Use Each

In practice, keep three kinds of guidance distinct. Only the first two are Claude Code features; the third is just good prompting hygiene. Confusing them is a common source of bloated CLAUDE.md files and under-used skills.

  • CLAUDE.md — always-loaded facts: project conventions, tech choices, constraints, and recurring preferences. Claude reads this on every session, so keep it lean.
  • Skills — reusable workflows and domain playbooks. Claude keeps a listing entry for available skills so it knows when they are relevant; the full skill body loads only when the skill is invoked. One important exception: if you set disable-model-invocation: true, the description is not kept in context until you invoke the skill yourself.
  • One-off prompts — ad hoc instructions for a task you may never repeat. Useful, but not a separate persistent Claude Code mechanism.

Older Claude Code material may talk about custom commands as a separate feature. Today, custom commands have been merged into skills: a file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md can both create /deploy, but skills are the more capable format.

If you have pasted the same playbook three times, it belongs in a skill. If a section of CLAUDE.md has grown into a procedure rather than a fact, extract it.

2. Anatomy of a Skill

Every skill is a directory. The only required file is SKILL.md.

~/.claude/skills/my-skill/
├── SKILL.md          # required — instructions and frontmatter
├── reference.md      # optional — detailed lookup material
├── template.md       # optional — output template
└── scripts/
    └── helper.py     # optional — script Claude can execute

In practice, most SKILL.md files use YAML frontmatter plus a markdown body. Anthropic's current docs make all frontmatter fields optional, but the frontmatter is where discovery, invocation control, and tool pre-approval live, so you should usually include it.

---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when
  explaining how code works, teaching about a codebase, or when the
  user asks "how does this work?"
---

When explaining code, always:

1. **Start with an analogy** — compare the concept to everyday life
2. **Draw an ASCII diagram** — show the flow or structure visually
3. **Walk through it step by step** — no implementation assumptions
4. **Flag one common misconception** — what do people usually get wrong?

The name becomes the /slash-command; if you omit it, Claude uses the directory name. The description is what Claude reads to decide whether to load the skill automatically; if you omit it, Claude falls back to the first paragraph of the markdown body. The combined description and when_to_use listing text is truncated to 1,536 characters.

3. Where Skills Live

Where you store a skill determines who gets it.

  • Enterprise-managed — deployed to all users in your organisation
  • ~/.claude/skills/skill-name/ — personal, available across all your projects
  • .claude/skills/skill-name/ in a repo — project-scoped, committed to version control
  • Plugin skills — bundled with an installed plugin and available where that plugin is enabled

When the same skill name exists at enterprise, personal, and project levels, higher-priority wins: enterprise beats personal beats project. Plugin skills use a namespace like plugin-name:skill-name, so they do not conflict with those levels. Claude Code also auto-discovers nested .claude/skills/ directories in monorepos, and legacy .claude/commands/ files still work.

4. Invocation Control

By default, both you and Claude can invoke any skill. Two frontmatter fields change that.

disable-model-invocation: true

Only you can trigger this skill. Use it for anything with side effects: /deploy, /commit, /send-report. You do not want Claude deciding to deploy because the code looks ready.

---
name: deploy
description: Deploy the application to production
disable-model-invocation: true
---

Deploy $ARGUMENTS to production:
1. Run the full test suite
2. Build the application
3. Push to the deployment target
4. Confirm the deployment is healthy

user-invocable: false

Only Claude can trigger this skill. Use it for background knowledge that is not a command — context Claude should apply automatically, but that /legacy-system-context is not a meaningful action for you to take.

5. Arguments

Both you and Claude can pass arguments when invoking a skill. Use $ARGUMENTS for the full argument string, $ARGUMENTS[N] for a 0-based position, or the shorthand $0, $1 for individual values.

---
name: fix-issue
description: Fix a GitHub issue by number
disable-model-invocation: true
---

Fix GitHub issue $ARGUMENTS following our coding standards.

1. Read the issue description
2. Understand the requirements
3. Implement the fix
4. Write tests
5. Create a commit

Running /fix-issue 247 gives Claude "Fix GitHub issue 247 following our coding standards." If you pass arguments but the skill body does not reference $ARGUMENTS, Claude Code appends ARGUMENTS: 247 to the rendered skill so the input is still preserved.

6. Dynamic Context Injection

Skills can run shell commands before the content reaches Claude. Use the !`command` syntax. The output replaces the placeholder — Claude receives actual data, not the command itself.

---
name: pr-summary
description: Summarise the current pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- Diff: !`gh pr diff`
- Comments: !`gh pr view --comments`
- Changed files: !`gh pr diff --name-only`

## Your task
Summarise this pull request for a reviewer who has not seen it before.

When this skill runs, each !`...` block executes first. Claude receives the fully-rendered prompt with the actual PR data already inserted. This is preprocessing, not something Claude executes.

For multi-line commands, use a fenced code block opened with ```!. If you need to disable shell injection for user, project, plugin, or additional-directory skills, Anthropic documents a disableSkillShellExecution setting; bundled and managed skills are not affected.

7. Pre-Approving Tools

The allowed-tools field grants Claude permission to use specific tools without prompting you each time the skill is active. It does not restrict anything — all tools remain callable, and your normal permission settings govern the rest.

---
name: commit
description: Stage and commit the current changes
disable-model-invocation: true
allowed-tools: Bash(git add *) Bash(git commit *) Bash(git status *)
---

If you actually want to block tool use, do that in Claude Code's permission settings with deny rules; allowed-tools is an allowlist for approval flow, not a sandbox.

8. Subagent Execution

Add context: fork to run a skill in an isolated subagent. The skill content becomes the task prompt. The subagent does not see your conversation history, but it does run with the selected agent type's system prompt and your CLAUDE.md guidance. Results are summarised and returned to your main session.

---
name: deep-research
description: Research a topic thoroughly in an isolated context
context: fork
agent: Explore
---

Research $ARGUMENTS thoroughly:

1. Find relevant files using Glob and Grep
2. Read and analyse the code
3. Summarise findings with specific file references

The agent field sets the execution environment — model, tools, and permissions. Built-in options include Explore, Plan, and general-purpose; you can also point to a custom subagent. If omitted, general-purpose is used.

9. Supporting Files

Keep SKILL.md focused. Move large reference material — API docs, detailed schemas, extended examples — to separate files in the same directory. Claude loads them only when the task needs them.

my-skill/
├── SKILL.md          # overview and navigation
├── reference.md      # detailed API docs — loaded when needed
├── examples.md       # usage examples — loaded when needed
└── scripts/
    └── validate.sh   # executed, not loaded into context

Reference supporting files from your SKILL.md so Claude knows what each contains and when to read it. Scripts are especially efficient: when Claude runs validate.sh, only the output enters the context window — the script code itself does not. Anthropic's guidance is to keep SKILL.md under roughly 500 lines and push bulky detail into supporting files.

10. Context Lifecycle and Compaction

When a skill is invoked, the rendered SKILL.md enters the conversation as a message and stays there for the rest of the session. Claude does not re-read the skill file on later turns. Write guidance that should apply throughout a task as standing instructions, not one-time steps.

After auto-compaction, Claude Code re-attaches the most recently invoked skills: up to 5,000 tokens each, 25,000 tokens combined. Older skills can be dropped if many have been invoked. If a skill seems to stop influencing behavior after compaction, re-invoke it.

11. Bundled Skills

Claude Code currently documents several bundled skills. They use the same prompt-based mechanism as the skills you write yourself, not a separate hardcoded system, so they are good examples to study.

  • /simplify — reviews recently changed files for reuse, quality, and efficiency issues, then fixes what it finds
  • /debug — enables debug logging for the current session and investigates issues using the session debug log
  • /loop — reruns a prompt on an interval while the session stays open
  • /batch — applies a task across multiple files in parallel
  • /claude-api — loads Claude API reference material and SDK guidance for your project's language

Type /simplify after making a set of changes to see how a well-crafted skill prompt reads.

Check

If you can create a skill, control whether Claude or you invokes it, pass arguments to it, and explain the difference between a skill body and its description — you are ready to start building your own skill library.
