Plan for Expanding the Course Architecture Beyond Claude Code

March 24, 2026 · Target new subject: OpenAI Codex · Scope: plan the multi-subject architecture and prove it with one Codex module

Why the Structure Needs to Change

The current project works well for a single subject, but it is optimized around one course living in a flat namespace. Module numbering, syllabus references, and embedded course data all assume one course. That becomes brittle as soon as the repository needs multiple subjects.

  • course.html and index.html embed Claude-specific module metadata directly in page scripts.
  • File names like module-1-lesson.html assume there is only one module 1.
  • The shell has no concept of a subject, only a single module list.

Recommended Direction

Phase 1: Prove the Shape Additively

Keep the existing Claude course working as-is. Add Codex artifacts with explicit prefixes and use them to validate voice, structure, and fit.

Phase 2: Introduce a Shared Subject Manifest

Once the second subject feels right, move course structure data out of inline JavaScript and into subject-level manifests.

courses/
  claude-code/
    course.json
    syllabus.html
    modules/
      01/
        lesson.html
        index.html
        exercise.html
  openai-codex/
    course.json
    syllabus.html
    modules/
      01/
        lesson.html
        index.html

Phase 3: Convert the Course Shell Into a Subject Shell

Refactor the shell so it can load one selected subject rather than assuming Claude. That shell should support a subject picker, subject-scoped progress tracking, and manifest-driven rendering of modules and resources.

Phase 4: Standardize Resource Taxonomy

The course should treat lessons, exercises, references, quizzes, diagrams, slides, and tools as resource types, not hardcoded assumptions per module.

Important Presentation Change

Learner-facing content should be delivered as responsive HTML and SVG, not as Markdown that depends on a browser extension. Markdown can remain a drafting format, but it should not be the presentation layer the learner relies on.

Proposed Codex Course Shape

  1. Orientation — What Codex Is and Where It Runs
  2. AGENTS.md — Durable Guidance for Codex
  3. Prompting and Planning
  4. Permissions, Sandboxing, and Trust Boundaries
  5. Working Inside VS Code
  6. Review, Testing, and Git Checkpoints
  7. Config and Personal Defaults
  8. MCP and External Context
  9. Skills and Reusable Workflows
  10. Automations and Longer-Running Work
  11. Codex Cloud and Remote Delegation
  12. Capstone

What This First Codex Pass Should Prove

  • The course voice still works for a different tool.
  • The app and VS Code extension fit the existing exercise-driven teaching style.
  • The repo should become a family of subjects, not one Claude-specific structure.
  • HTML-first delivery is the right long-term presentation model.

Recommended Next Step

If this direction feels right, the next implementation pass should create the courses/ structure, define the first manifest format, and migrate one shell page to render from manifest data before moving the whole Claude course.
