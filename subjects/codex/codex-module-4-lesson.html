Module 4: Permissions and Sandboxing

Working Effectively with OpenAI Codex — A Practitioner's Course

Why This Module Matters

Codex becomes useful when you let it act. It becomes safe when you control the boundary of that action. The current docs make an important distinction here: sandboxing and approvals are not the same control.

The sandbox defines what Codex can technically do without escaping its boundary. The approval policy defines when Codex must stop and ask before going beyond that boundary.

1. Sandbox and Approval Policy Are Different Controls

Think of sandboxing as the fence and approval policy as the checkpoint. The fence defines the allowed area. The checkpoint decides when the agent must pause for a human decision.

Sandboxing defines the boundary. Approval policy defines when Codex pauses.

If you blur those together, mode selection feels confusing. If you keep them separate, the model becomes much easier to reason about.

2. The Sandbox Applies to Commands Too

OpenAI’s docs are explicit that the sandbox does not just affect built-in file edits. If Codex runs local commands like git, test runners, or package managers, those commands inherit the same boundaries.

That matters because a task may look small in chat while still requiring command execution that crosses the current boundary.

3. The Common Sandbox Modes

  • read-only: good for inspection and questions
  • workspace-write: the normal low-friction local mode for reading, editing in the workspace, and routine local work
  • danger-full-access: removes the sandbox boundary and should be used only intentionally

The practical default for local work is usually workspace-write, not full access.

4. The Common Approval Policies

  • untrusted: ask before running commands outside the trusted set
  • on-request: work inside the sandbox normally, ask when a task needs to cross the boundary
  • never: do not stop for approval prompts

never is not a convenience toggle. It changes the trust model.

5. Full-Auto Is Not Full Access

This distinction matters because people often compress the two. OpenAI’s docs say --full-auto is the lower-risk local automation preset: workspace-write plus on-request. That is not the same as unrestricted access.

True full access is danger-full-access combined with never. That removes both the sandbox boundary and the approval pauses.

6. How You Control It

In the Codex app and IDE, the current docs say you use the permissions selector under the composer or chat input. In the CLI, you can switch modes during a session with /permissions.

The operating habit you want is deliberate switching, not leaving Codex permanently over-permissioned.

7. Network, Outside-Workspace Writes, and Protected Areas

The docs also make two operational points that are easy to miss:

  • network access may still require explicit allowance or approval even in workspace-write
  • some environments keep directories like .git/ and .codex/ read-only even when the rest of the workspace is writable

That is why a command like git commit can still need approval even when editing files feels permitted.

8. Prefer Narrow Exceptions Over Broad Access

The docs recommend using writable roots and rules when you need a specific exception. That is usually better than jumping to broad access because it keeps the trust boundary legible.

The safest operational heuristic is simple: widen access only as much as the task requires, and only for as long as it requires.

9. What Good Judgment Looks Like

For most tasks, the right move is not “full access or nothing.” It is one of these:

  • read-only when you are inspecting or learning
  • workspace-write when you need normal local edits
  • network opt-in or writable-root adjustments when the task has a specific boundary need
  • full access only when you intentionally want to remove those boundaries

Check

If you can explain the difference between sandboxing and approvals, distinguish full-auto from full access, and choose a narrower mode before a broader one, you are ready for VS Code workflow details.
