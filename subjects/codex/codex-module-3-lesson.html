Module 3: Prompting and Planning

Working Effectively with OpenAI Codex — A Practitioner's Course

Why This Module Matters

Codex is strong enough that a vague request often works. That is useful, but it can mislead you. The docs make a sharper distinction: simple tasks can be direct, but complex or ambiguous tasks should be planned before implementation starts.

The practical skill is not “write perfect prompts.” It is matching the prompt shape to the task shape.

1. Not Every Task Needs Planning

If the task is narrow, local, and easy to verify, a direct prompt is often enough. You do not need planning theater for a small rename, a targeted explanation, or a tightly bounded edit.

What matters is whether the task can be described clearly in one pass and whether success is easy to recognize.

2. Plan First When the Task Is Hard to Describe Well

OpenAI’s Codex docs explicitly recommend planning first for tasks that are complex, ambiguous, or hard to describe well. That is the inflection point. If you find yourself packing caveats, edge cases, and half-formed goals into one giant message, you probably want a plan-first interaction.

Plan mode is one route. Another is to ask Codex to interview you first, challenge your assumptions, and turn the fuzzy request into a more concrete plan before coding starts.

3. A Good Prompt Defines the Working Frame

For Codex, a strong request usually does four things:

  1. states the goal
  2. gives the relevant repo or task context
  3. defines constraints
  4. defines what done means

This is similar to general prompting advice, but it matters more with an agent because the task can fan out into tool use, file changes, and verification steps.

4. Clarifying Questions Are a Feature

When the task is ambiguous, clarifying questions are not resistance. They are part of planning. Good agentic work depends on reducing the chance of solving the wrong problem efficiently.

If you would interrupt a junior engineer to clarify the request, you should expect Codex to do the same.

5. Done-When Is What Prevents Wandering

A prompt without completion criteria invites partial success. Codex might do something sensible and still miss what you actually needed because you never told it what would count as finished.

Useful done-when statements are concrete:

  • bug reproduced and fixed with a regression test
  • new page matches the existing visual language and passes the test suite
  • CLI command succeeds and docs are updated

6. Durable Guidance and One-Off Guidance Work Together

AGENTS.md handles durable repo truth. Your prompt handles the current task. Problems happen when you mix those roles. If the prompt keeps carrying permanent project rules, the repo is under-instructed. If AGENTS.md tries to encode every one-off project, the repo becomes bloated.

7. The Planning Ladder

You can think of Codex requests as a ladder:

  • direct request for small, obvious work
  • direct request plus constraints for bounded implementation
  • plan-first request for ambiguous or multi-step work
  • structured execution plan for larger efforts

The mistake is staying on the bottom rung after the task has already become murky.

8. What Good Prompting Looks Like in Practice

Good prompting with Codex is less about ornate language and more about operational clarity. Give the task a shape that the agent can execute and verify. If the task is fuzzy, ask for planning first. If it is sharp, stay direct. If it keeps missing in the same way, improve the durable guidance around it.

Check

If you can distinguish direct tasks from plan-first tasks, welcome clarifying questions when they are needed, and define clear done-when criteria, you are ready for permissions and sandboxing.
