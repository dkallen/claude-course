# ADR-001: Custom-Built Course Platform Over Existing LMS

**Date:** 2026-03-29
**Status:** Accepted
**Deciders:** David Allen

---

## Context

The course platform was originally built as a custom static HTML/CSS/JS application for personal learning. As the platform matured (6 fully built modules across 2 subjects, custom interactive tools, phase-based learning model), two growth vectors emerged:

1. **Expanding topics** — adding new AI tool subjects beyond Claude Code and OpenAI Codex.
2. **Offering to others** — moving from personal use to a multi-user training platform.

This raised the question of whether to migrate to an established Learning Management System (Teachable, Thinkific, Moodle, Open edX, LearnDash, etc.) or continue building on the custom platform.

A formal requirements analysis was conducted (see [requirements.md](../../requirements.md)) and the following tensions were evaluated.

## Tensions Evaluated

1. **Authoring speed vs. creative control.** An LMS provides a WYSIWYG editor for lessons and quizzes but reduces fine-grained control over layout, interaction design, and visual presentation. The custom platform allows pixel-level control.

2. **Self-directed vs. guided learning model.** The current design is radically self-directed — no gating, manual checkboxes, advisory self-assessment "checks." Most LMS platforms default to gated progression and scored assessments. Configuring that philosophy out of an LMS creates ongoing friction.

3. **Custom tools as first-class content vs. embedded widgets.** Bespoke interactive tools (CLAUDE.md Builder, Prompt Calibrator, Prompt Spectrum, Spec-to-Tests Workbench) are equal citizens in the current resource list. In an LMS, they would be iframes or SCORM packages — technically functional but second-class in navigation and progress tracking.

4. **Phase-based module structure.** The "Phase 1 → Phase 5" grouping within each module is more nuanced than most LMS "lesson sequence" models. Flattening to a simple ordered resource list would change the learning experience.

5. **Speed of experience.** The current platform is instant — no loading states, no server round-trips for navigation. An LMS introduces network latency for every page transition.

## Decision

**Stay with the custom-built platform.** Do not migrate to an existing LMS.

## Rationale

1. **AI-assisted authoring changes the cost equation.** Content is not hand-coded — it is generated by AI (Claude). AI can produce HTML files directly far more efficiently than it can drive a proprietary LMS content editor or API. Existing LMS platforms were built before AI-assisted authoring was practical, and their content pipelines add friction rather than removing it.

2. **Custom interactive tools are the differentiators.** The builders, calibrators, and workbenches are purpose-built React applications that would become second-class citizens embedded as iframes or SCORM packages inside an LMS. Keeping them as first-class resources preserves the learning experience.

3. **AI-in-the-course is a planned feature.** Future plans include embedding AI directly into course content — essay grading, adaptive tutoring, and other capabilities not yet designed. This is straightforward in custom HTML/JS but would require fighting LMS plugin architectures or paying for expensive AI-enabled LMS tiers that may not offer sufficient flexibility.

4. **Pedagogical philosophy conflicts with LMS defaults.** The self-directed, no-gating design runs counter to how most LMS platforms are architected. Working against the grain of the platform creates ongoing maintenance cost.

5. **LMS platforms may evolve AI features, but at a price.** Platforms that add AI capabilities will charge premium pricing for them, while the custom platform can implement equivalent features at the cost of development time — which is itself AI-assisted.

## Consequences

### Positive
- Full creative control over content presentation, interaction design, and learning model.
- AI-assisted authoring workflow remains frictionless (generate HTML, add to course-data.js).
- No platform vendor lock-in or recurring licensing costs.
- Freedom to embed AI features directly as they become relevant.
- Instant navigation performance preserved.

### Negative
- **Multi-user capabilities must be built from scratch.** Authentication, server-side progress storage, enrollment, and access control are unsolved. These are standard LMS features that now require custom development.
- **Analytics must be built.** Completion rates, drop-off analysis, and engagement metrics are not available out of the box.
- **Payment integration must be built** if the course is offered commercially.
- **Ongoing platform maintenance** is the author's responsibility — there is no vendor handling security patches, browser compatibility, or accessibility compliance.

### Risks
- Scope creep: building multi-user infrastructure could consume time better spent on content.
- The custom platform may not scale gracefully under load if offered to many concurrent users.
- Accessibility compliance (WCAG, Section 508) must be self-audited rather than inherited from a certified platform.

### Mitigations
- Prioritize content expansion over platform features in the near term.
- When multi-user features are needed, evaluate lightweight auth/storage solutions (e.g., Supabase, Firebase) rather than building from scratch.
- Use AI-assisted accessibility auditing as part of content generation workflow.
